# Rename Sequencing Checklist

> 只记录 rename inventory、风险分级与依赖顺序；不做任何实际改名。

## 1. 当前可立即安全改名候选

- 无。

## 2. 需等 contract 收紧后再改

| 候选 | 风险 | 依赖 |
| --- | --- | --- |
| 文档内的术语别名、示例名 | 中 | 先收紧对外表述，再统一文档词汇 |
| 内部实现符号但仍被少量调用的辅助命名 | 中 | 先确认调用面收敛，再改本地实现名 |

## 3. 需等 canonical module path / compatibility layer 收口后再改

| 候选 | 风险 | 依赖 |
| --- | --- | --- |
| 仍存在兼容导出的模块入口 | 高 | 先收口 canonical module path，再移除兼容层后改名 |
| 跨模块重导出的公开类型 / 函数名 | 高 | 先让唯一入口稳定，再做重命名 |
| 处在桥接层、适配层的实现文件名 | 高 | 先确认兼容层结束，再统一文件与符号名 |

## 4. 当前禁改类别

- 外部契约：HTTP 路径、请求/响应字段、错误码、版本号语义
- 跨边界语义：root / `santi` / `santi-link` / `santi-cli` 的职责边界名
- 已作为兼容层出口的公共符号
- 已被文档、脚本、测试、runbook 广泛引用的稳定名字
- 任何会改变 canonical module path 识别结果的名字

## 5. 最安全的执行顺序

1. 先冻结禁改类别，不碰外部契约与边界语义。
2. 再收口 canonical module path 与兼容导出，保证只有一条主路径。
3. 随后统一桥接层 / 适配层的内部实现名。
4. 最后处理文档里的术语别名和残留示例名。

## 6. 风险分级

- 低：仅文档层、无外部引用、无兼容层依赖
- 中：仍有少量内部引用，需先确认 contract 已收紧
- 高：涉及 public surface、compat layer、canonical path，必须后置
